The Atomic Operations

  • Store operations, which can have memory_order_relaxed, memory_order_release, or memory_order_seq_cst ordering
  • Load operations, which can have memory_order_relaxed, memory_order_consume, memory_order_acquire, or memory_order_seq_cst ordering
  • Read-modify-write operations, which can have memory_order_relaxed, memory_order_consume, memory_order_acquire, memory_order_release, memory_order_acq_rel, or memory_order_seq_cst ordering

Memory Orderings

Sequentially Consistent Ordering

Under sequentially consistent memory ordering, all threads see the same order of operations, or alternatively, all threads see the same global state. This is the default memory ordering for all atomic operations in C++ so does not need stating explicitly.

Consider the following program consisting of two threads. t1 writes to x and reads from y. t2 writes to y and reads from x. We then assert that both of the reads are not zero. 

#include <atomic>
#include <thread>
#include <cassert>

std::atomic<int> x{0}, y{0};
int r1 = 0, r2 = 0;

int main() {
    std::thread t1([] {
        x.store(1, std::memory_order_seq_cst);  // A
        r1 = y.load(std::memory_order_seq_cst); // B
    });

    std::thread t2([] {
        y.store(1, std::memory_order_seq_cst);  // C
        r2 = x.load(std::memory_order_seq_cst); // D
    });

    t1.join();
    t2.join();

    assert(!(r1 == 0 && r2 == 0));
}

The threads cannot both read zero and so the assertion never fails. If t1 writes to x before the read in t2, t2 must read that value, and vice versa if t2 writes to y before the read in t1. memory_order_seq_cst is the simplest memory order to reason about.

memory_order_seq_cst is the most expensive of all the memory orderings. This becomes of particular concern on non x86/64 hardware where there is no hardware level implementation for sequentially consistent ordering.

To clearly see why the assertion never fires, we can explicitly list all the possible global orderings of operations and the corresponding values of r1 and r2. No valid ordering results in both being zero.

Ordering r1 r2
A B C D01
A C B D11
A C D B11
C D A B10
C A D B11
C A B D11

Non-Sequentially Consistent Orderings

Relaxed Ordering

We now turn to the strange world of non-sequentially consistent memory ordering, where different threads don't have to agree on a global order of operations. We'll use the same example as before, but with memory_order_relaxed.

In our nice mental model of concurrent programming, all operations are neatly interleaved so you could in theory write them all down as a single global order of operations. Throw this mental model out the window.

memory_order_relaxed is the least restrictive ordering. Atomic operations are still atomic, but there are no synchronization or visibility guarantees between threads. If thread A writes to a variable, there is no guarantee that thread B will see that write. On x86 we can generally rely on writes being seen eventually because of the strong memory model on x86.

#include <atomic>
#include <thread>
#include <cassert>

std::atomic<int> x{0}, y{0};
int r1 = 0, r2 = 0;

int main() {
    std::thread t1([] {
        x.store(1, std::memory_order_relaxed);  // A
        r1 = y.load(std::memory_order_relaxed); // B
    });

    std::thread t2([] {
        y.store(1, std::memory_order_relaxed);  // C
        r2 = x.load(std::memory_order_relaxed); // D
    });

    t1.join();
    t2.join();

    assert(!(r1 == 0 && r2 == 0));
}

This assertion can sometimes fail. How can r1 and r2 both read zero? What's actually happening is that t2 sees a different version of reality to t1. This is what we mean when we say there is no longer a single global order of operations.

t1 writes to x and then reads from y. When t1 reads from y, it can read an older value even if t2 has already written to y. This is possible because each core can buffer its own stores.

Even though in t1 the store to x happens before the load from y in program order, there is no guarantee that t2 will observe the store to x before performing its load. Likewise for t1 observing t2's store.

The only requirement is that if a thread sees a particular value of an atomic variable, it can't then see a previous value on subsequent reads.

Acquire-Release Ordering

Acquire-release sits somewhere between relaxed and sequentially consistent in terms of the guarantees it offers.

Reads use acquire, stores use release; read-modify-write operations can be either. An acquire should be paired with a release. Synchronization is between the two threads that hold the acquire-release pair — other threads don't sync.

#include <atomic>
#include <thread>
#include <cassert>

std::atomic<bool> ready{false};
int data = 0;

void producer() {
    data = 42;
    ready.store(true, std::memory_order_release); // publish
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {
        ; // spin
    }
    assert(data == 42); // guaranteed
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join();
    t2.join();
}

This is all well and good, but we could have just used sequentially consistent ordering here and got the same outcome. To see how the behavior differs, we add more threads.

#include <atomic>
#include <thread>
#include <cassert>

std::atomic<bool> x{false};
std::atomic<bool> y{false};
int r1 = 0;
int r2 = 0;

void t1() { x.store(true, std::memory_order_release); }
void t2() { y.store(true, std::memory_order_release); }

void t3() {
    while (!x.load(std::memory_order_acquire)) {}
    r1 = y.load(std::memory_order_acquire);
}

void t4() {
    while (!y.load(std::memory_order_acquire)) {}
    r2 = x.load(std::memory_order_acquire);
}

int main() {
    std::thread a(t1), b(t2), c(t3), d(t4);
    a.join(); b.join(); c.join(); d.join();
    assert(r1 != 0 && r2 != 0);
}

The assert can fire here. r1 and r2 can both be zero. t1 syncs with t3; t2 syncs with t4. There is no synchronization between t2 and t3, or t1 and t4. Synchronization happens only on the acquire-release pair between two threads.

Acquire-Release and Non-Atomic Operations

Acquire-release can also be used to synchronize non-atomic operations.

#include <atomic>
#include <thread>
#include <cassert>

int data{0};
std::atomic<bool> ready{false};

void producer() {
    data = 42;                                        // non-atomic write
    ready.store(true, std::memory_order_release);     // publish
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {} // spin
    assert(data == 42); // all writes before release are now visible
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join();
    t2.join();
}

There is no data race on data because the consumer must wait until the producer signals ready. If the producer updates data many times, use a relaxed atomic — the release-acquire will synchronize those relaxed writes such that any relaxed write before the release in the producer is visible after the acquire in the consumer.
