Why Lockfree

The typical motivations given for using lockfree programming are as follows:

  • No lock contention — threads don't wait to acquire a lock. Any contention happens only on a single atomic variable, which is usually much cheaper than acquiring a lock as it avoids kernel involvement (sleeping, waking, context switching etc.).
  • Progress guarantees — at least one thread will always be making progress. With locks, we have issues such as deadlocking, priority inversion, and convoying (slow threads holding up other threads).
  • Scaling concurrency — without large critical sections where only one thread can do useful work, lock-free programming can scale better with more concurrency.

These advantages are all well and good, however there are some domains that live and die by millisecond latency. If latency spikes kill your application, lock-free data structures are your friend.

Execution speed matters in domains such as HFT.

Treiber Stack

The Treiber stack uses compare-and-swap to push and pop nodes.

push

Push is essentially a compare-and-swap on the old head and new head.

std::atomic<Node*> head{nullptr};

void push(int value) {
    Node* new_node = new Node{value, nullptr};

    Node* old_head;

    old_head = head.load();
    new_node->next = old_head;
    head.compare_exchange_strong(old_head, new_node);
}

pop

Pop is a compare-and-swap on the current head and head->next. If head hasn't changed during the CAS, set head->next as head and return the old head.

Node* pop() {
    Node* old_head;
    Node* new_head;

    old_head = head.load(std::memory_order_acquire);
    if (old_head == nullptr)
        return nullptr;
    new_head = old_head->next;
    head.compare_exchange_strong(old_head, new_head,
        std::memory_order_acquire,
        std::memory_order_relaxed);
    return old_head;
}

The ABA Problem

Compare-and-swap checks pointers, not values. This can lead to a subtle corruption:

1. Thread 1 begins a pop for node A (the current head) and is paused just after new_head = old_head->next.

2. Thread 2 pops node A and node B, then pushes a new node to the same memory address as A.

3. Thread 1's CAS succeeds — head is still at A — so it pops A and makes B the new head.

B was already freed by Thread 2. The stack is now in a corrupted state.

Three common solutions:

Tagged Pointers

Perform CAS on a tagged pointer and increment the tag every time the head node is updated. This ensures the pointer is unique even if it addresses the same memory.

struct TaggedPtr {
    Node*    ptr;
    uint64_t tag;
};

The tagged pointer is now 128 bits wide, requiring double-width CAS — or the tag can be packed into a single word. Tagged pointers don't prevent use-after-free; freed memory can still be dereferenced. In practice, hazard pointers are preferred.

Hazard Pointers

Keep a record of "in-use" pointers and never free a pointer that is still in use. This solves the use-after-free issue with ABA. On pop, the node is retired (added to a retire list) rather than freed immediately. The retire list is periodically scanned and nodes not currently in use are reclaimed.

Epoch-Based Reclamation

Maintain a global epoch counter. Once all threads have entered the current epoch, advance it. When a thread is done with a node, add it to the retire list along with the current epoch. Any node ≥2 epochs old is safe to delete — no other thread can be referencing it. Check for deletable nodes each time the retire list reaches a threshold size.

* * *

Michael Scott Queue

A FIFO queue with support for multiple producers and multiple consumers.

Maintains head and tail pointers and uses a dummy node at the front to simplify enqueue/dequeue logic. The tail pointer may not always point to the very last node — a deliberate design choice that significantly improves throughput.

SPSC Ring Buffer

For the single-producer single-consumer case, release/acquire semantics with load and store are sufficient.

False Sharing

Apple Silicon uses 128-byte cache lines. Using alignas(128) ensures that full and slot do not land on the same cache line. If they did, the CPU would have to synchronize access from different cores, adding cycles. This is known as false sharing.

struct RingBufferLockFree {
    alignas(128) std::atomic<bool>     full{false};
    char pad1[128 - sizeof(std::atomic<bool>)];

    alignas(128) std::atomic<uint64_t> slot{};
    char pad2[128 - sizeof(std::atomic<uint64_t>)];

    void push(uint64_t v) {
        while (full.load(std::memory_order_acquire))
            CPU_PAUSE();
        slot.store(v, std::memory_order_relaxed);
        full.store(true, std::memory_order_release);
    }

    bool pop(uint64_t& out) {
        if (!full.load(std::memory_order_acquire))
            return false;
        out = slot.load(std::memory_order_relaxed);
        full.store(false, std::memory_order_release);
        return true;
    }
};

Benchmark

This simple benchmark demonstrates the latency difference between lock-free and locking implementations. One thread pushes a timestamp; the other pops it and records the round-trip time.

std::thread prod([&]() {
    while (run.load()) rb.push(now_ns());
});

std::thread cons([&]() {
    while (run.load()) {
        uint64_t ts;
        if (rb.pop(ts)) samples.push_back(now_ns() - ts);
    }
});
